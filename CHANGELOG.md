# Changelog

All notable changes to this project are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

## [Unreleased]

### Added

- Runtime-configurable runner UID/GID via `RUNNER_UID` and `RUNNER_GID` env vars. The base image entrypoint now starts as root, optionally remaps the in-container `runner` user to a caller-specified UID/GID (recursive chown of `/home/runner`, `/runner`, `/opt/herd` happens once, on the first start with a new UID), then drops privileges via `gosu` before invoking the GitHub Actions runner. Lets operators match host UID/GID for bind mounts (e.g. TrueNAS SCALE's `apps` user at 568:568) without rebuilding the image — set `RUNNER_UID=568` / `RUNNER_GID=568` in `.env` and restart. The build-time default is `1001:1001` (the historical default — `ubuntu:24.04` reserves UID 1000 for its `ubuntu` user, so previous images already ran runner at 1001 even though that was implicit). Pinning it explicitly preserves backward compat with codex-auth volumes and other bind mounts owned by 1001 on existing hosts. **Migration:** to opt in, ensure `Dockerfile.herd_runner` does **not** end with `USER runner` (older `herd init`-generated wrappers added this); the entrypoint detects a non-root start and skips the remap entirely to preserve backward compatibility. `RUNNER_UID=0` / `RUNNER_GID=0` are rejected — the GitHub Actions runner refuses to run as root.
- Codex provider (`agent.provider: codex`) — shells out to the OpenAI Codex CLI with API-key auth. `OPENAI_API_KEY` is auto-mapped to `CODEX_API_KEY` at invocation time when `CODEX_API_KEY` is unset; an explicit `CODEX_API_KEY` always wins. The worker workflow and docker exec forward both secrets. `agent.model` takes a **bare** model ID (e.g. `gpt-5-codex`, `gpt-5.2`), not a provider-prefixed form. New `agent.codex_reasoning_effort` config field (`minimal` | `low` | `medium` | `high`, default `medium`) maps to `-c model_reasoning_effort=<value>` on every Codex invocation.
- `herd codex doctor` reports local Codex CLI/auth/config diagnostics, including effective auth path and stale `CODEX_AUTH_JSON` notices.
- Codex ChatGPT-subscription auth (opt-in alternative to API-key auth). Personal subscriptions (Plus/Pro/Team/Edu) authenticate via a single base64-encoded `CODEX_AUTH_JSON`, with parallelism via `docker compose up -d --scale worker=N` (all worker containers share one `codex-auth` docker volume); ChatGPT Enterprise authenticates via an agent-identity JWT in `CODEX_ACCESS_TOKEN`. Provisioning seeds the docker-volume-backed `~/.codex` from the base64 env var, records a `.herd-seed` marker, and re-seeds automatically when the env value changes (no force-seed flag). A `herd codex keepalive-loop` background daemon (spawned by the runner entrypoint when `CODEX_AUTH_JSON` is set) keeps idle OAuth chains warm via a near-noop `codex exec` on a default 6-day cadence, tunable via `HERD_CODEX_KEEPALIVE_INTERVAL`.
- Interactive `herd plan` for the Codex provider — running `herd plan` with no arguments now opens a conversational Codex session to refine the brief before decomposition (parity with the `claude` and `opencode` providers), then captures the structured plan via a file passed through the `HERD_PLAN_OUT` env var (with the JSON Schema path passed through `HERD_PLAN_SCHEMA`). `herd plan "<description>"` keeps the existing headless `codex exec` behavior.
- Published GHCR runner base image at `ghcr.io/herd-os/herd-runner-base` — public, multi-arch (linux/amd64, linux/arm64), version-pinned to the herd release.
- `herd image build` and `herd image publish` commands to build and push a customized runner image to `ghcr.io/<owner>/<repo>-herd-runner`.
- `.github/workflows/herd-publish-runner.yml` workflow that builds and pushes the consumer runner image on push to `Dockerfile.herd_runner` on `main` AND on `workflow_dispatch` (gated on `HERD_ENABLED`, requires `packages: write`). For `herd-os/herd` itself, the tag-driven `publish-runner-image` job in `release.yml` ALSO rebuilds the wrapper on every release; in the rare case where a release tag closely follows a wrapper-touching PR, both jobs fire and produce a duplicate build. That double-build is herd-os/herd-specific and acceptable.

### Changed

- `herd init` no longer generates `Dockerfile.herd_runner_base`; an existing one is removed and the base service is dropped from `docker-compose.herd.yml`.
- `Dockerfile.herd_runner` now uses `FROM ghcr.io/herd-os/herd-runner-base:<version>` (pulled from GHCR) instead of the locally built `herd-runner-base`.
- `herd init` no longer generates `entrypoint.herd.sh` in consumer repos — the entrypoint is now baked into the published base image `ghcr.io/herd-os/herd-runner-base`. A leftover copy from an older init is removed automatically on the next `herd init`.
- AI provider auth env vars (`ANTHROPIC_API_KEY`, `CLAUDE_CODE_OAUTH_TOKEN`, `OPENAI_API_KEY`, `CODEX_API_KEY`, `CODEX_ACCESS_TOKEN`, `GEMINI_API_KEY`) are no longer surfaced from GitHub Actions secrets in the worker workflow template — they live only in the runner's `.env` (injected by `docker-compose`), matching the architecturally-recommended setup. A workflow `env:` block overrides container env unconditionally, so an unset secret would clobber the real `.env` value at the step level; removing the secrets path eliminates that footgun (generalizing the `CODEX_AUTH_JSON` protection from #694). **Migration:** users who previously configured these values **only** as GitHub Actions secrets (without `.env`) must add the same values to the runner's `.env` for workers to authenticate. `HERD_GITHUB_TOKEN` and `workers.extra_env` secrets are unaffected.
- Agent settings now support sparse per-role overrides under `agent.planner` and `agent.workers` for `provider`, `binary`, `model`, `max_turns`, `codex_reasoning_effort`, and `codex_sandbox`; `agent.exec` and `agent.exec_image` remain base-only. Unset Codex sandbox values resolve from role and execution context: local planning stays empty so Codex uses `workspace-write`, while Docker planning plus worker, integrator, and review runs default to `danger-full-access`. **Migration:** configs with an explicit base `agent.codex_sandbox` keep that behavior. Users who previously set broad base `agent.codex_sandbox: danger-full-access` may keep it, or move it to `agent.workers.codex_sandbox` if they want safer local planning.

### Fixed

- Consumer repos using the `herd-publish-runner.yml` template now auto-rebuild their customized runner image when an `Update HerdOS to <tag>` PR merges, restoring behavior that #713 inadvertently removed. The template had its `push: paths: [Dockerfile.herd_runner]` trigger dropped in #713 to avoid a duplicate build in `herd-os/herd`'s own release flow (where `release.yml`'s `publish-runner-image` job ALSO rebuilds the wrapper on tag), but consumer repos have no `release.yml` and thus had no auto-rebuild path at all. Workers were silently running with stale baked-in agent CLIs (codex, claude, opencode) and stale project-specific tools after every herd upgrade unless a maintainer manually ran `gh workflow run herd-publish-runner.yml`. The trigger is restored; the rare double-build in `herd-os/herd` is the accepted trade-off.
- Codex provider no longer silently overrides ChatGPT-subscription auth with `OPENAI_API_KEY`. The `OPENAI_API_KEY` -> `CODEX_API_KEY` convenience mapping is now skipped when a subscription `auth.json` exists (under `$CODEX_HOME`, or `~/.codex`), so a stray `OPENAI_API_KEY` in the shell can no longer clobber the subscription and bill against API quota. An explicit `CODEX_API_KEY` still always wins; pure API-key users (no `auth.json`) keep the mapping.
- Codex keepalive log destination moved from `/var/log/herd-codex-keepalive.log` to `/opt/herd/herd-codex-keepalive.log`. The runner user owns `/opt/herd/` (chowned at image build time and re-chowned by the `RUNNER_UID` remap), but does not own `/var/log/` — under the unprivileged runner introduced in #712, the previous redirect failed with `Permission denied` and the keepalive process **never started**, so the OAuth chain lapsed silently after ~8 days idle on subscription-auth setups. Pure API-key and Enterprise (`CODEX_ACCESS_TOKEN`) setups were unaffected (keepalive is gated off for those).
- Worker no longer silently labels an issue `herd/status:done` when the agent produces zero commits AND fails to emit the required structured no-op verdict block (`Findings reviewed against the current code:` ... `Conclusion:`). Provider-agnostic: surfaced during the Codex-on-TrueNAS dogfood (issues #729/#730/#731) where Codex's bubblewrap sandbox blocked every file read and `apply_patch` with "No permissions to create a new namespace" — the worker exited cleanly with no diff and the issues flipped to `done` despite zero work happening. Workers now require the verdict block as positive proof of intent for the no-op success path; missing block returns an error so the deferred failure handler labels the issue `failed` and the monitor re-dispatches. Legitimate no-ops are unchanged (the worker prompt has always mandated the block).
- Codex CLI sandbox is configurable via `agent.codex_sandbox` in `.herdos.yml` (one of `read-only`, `workspace-write`, `danger-full-access`). Empty values are resolved by role and execution context; container worker-like runs default to `danger-full-access` instead of requiring users to set a global base override.

### Removed

- Multi-replica Codex ChatGPT-subscription auth (`agent.codex_replicas`, `CODEX_AUTH_JSON_1`..`N` env vars, per-replica `docker-compose.herd.yml` services and volumes). Subscription users now run a single `codex login`, set `CODEX_AUTH_JSON`, and scale parallelism via `docker compose up -d --scale worker=N` — all worker containers share a single `codex-auth` docker volume. The single-`CODEX_AUTH_JSON` + `CODEX_ACCESS_TOKEN` (Enterprise) + API-key paths are unchanged.
- OpenCode subscription authentication paths (ChatGPT/Codex via `opencode-openai-codex-auth` and Anthropic via the `opencode-claude-auth` OAuth bridge). The OpenCode provider now supports only plain API-key auth (`ANTHROPIC_API_KEY` for `anthropic/*` models, `OPENAI_API_KEY` for `openai/*` models). The `claude` provider's `CLAUDE_CODE_OAUTH_TOKEN` auth is unchanged.
- Codex ChatGPT-subscription auth via the base64-encoded `CODEX_AUTH_JSON` env var, the in-process `provisionCodexAuth` seeding path, and the `.herd-seed` marker file. Subscription users now authenticate interactively against the running worker container: `docker exec -it -u runner <worker-container> codex login --device-auth`, which writes `auth.json` directly into the persistent `codex-auth` volume (the source of truth). The runner keepalive is now gated on the presence of `~/.codex/auth.json` instead of the `CODEX_AUTH_JSON` env var. **Migration:** Existing users whose `codex-auth` volume already contains an `auth.json` keep working unchanged — the volume is the source of truth. New subscription setups: `docker exec -it -u runner <worker-container> codex login --device-auth`. The `CODEX_AUTH_JSON` env var in `.env` is silently ignored from this release on; remove it at your leisure.
