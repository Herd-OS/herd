# Issue #696 — Review fixes (cycle 2)

## Problem
Single-replica path (`codex_replicas == 1`) rendered the `worker` service with
`deploy: replicas: 3` AND one shared `codex-auth` volume. For the codex
subscription path (Path B) this meant 3 runner containers shared one `auth.json`
and each ran its own keepalive-loop, violating the documented
one-auth.json-per-runner invariant (runners.md:218) and contradicting
configuration.md:119 (codex_replicas default 1 = 1 runner).

## Fix
The single-replica `worker` service renders `deploy: replicas: 1` when provider
is `codex` (auth isolation), keeping `replicas: 3` for non-codex providers
(preserves prior-cycle default). Provider threaded through the render chain.
Codex users wanting N parallel runners use `agent.codex_replicas > 1` (Path C,
per-replica volumes + seeds). Gating on provider (not init-time env detection)
is reliable: `herd init` does not load `.env`, so `CODEX_AUTH_JSON` is usually
not visible to the init process.
